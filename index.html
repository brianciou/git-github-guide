01

分支命名的藝術

原文提到分支就像「不同的時間軸」，這個比喻非常好。但當你的時間軸越來越多，好的命名就變得很重要——就像你電腦裡的資料夾，如果全都叫「新資料夾」，三天後你自己也找不到東西。

生活比喻
分支命名就像幫便利貼寫標題。你不會只寫「東西」，你會寫「週五會議待辦」。分支名稱也一樣，要讓未來的你（或同事）一眼就知道這條時間軸在做什麼。

常見的命名規則

業界有一套約定俗成的「前綴」分類法，就像資料夾前面的標籤：

前綴 用途 範例
feature/ 新增一個功能 feature/dark-mode
fix/ 修復某個問題 fix/login-button-crash
hotfix/ 緊急修復（線上出事了！） hotfix/payment-error
design/ 設計相關的調整 design/new-landing-page
refactor/ 重構（功能不變，但結構更好） refactor/simplify-nav
experiment/ 試驗性的想法 experiment/ai-chatbot

命名的小規則

✓ 好的命名
feature/user-profile-page fix/image-not-loading design/homepage-redesign

用英文小寫、用連字號 - 分隔、簡潔但有描述性

✗ 不好的命名
New_Branch test123 brian的修改

避免中文、空格、大寫混用、沒有意義的名字

對 Claude Code 說

「幫我建立一個新分支，我要做深色模式的功能」
→ Claude Code 會幫你建立 feature/dark-mode 並切換過去

02

一個功能的完整生命週期

了解命名之後，來看看一個功能從開始到結束的完整流程。這是大多數團隊（甚至一個人的專案）會遵循的節奏：

功能分支的生命週期：從 main 分出去 → 開發 → 合併回來

用 Claude Code 走完這整個流程，你只需要這幾句話：

  1. 「幫我從 main 建立一個新分支，我要做暗色模式」
  2. （你開始開發，做了一些改動）
  3. 「幫我儲存目前的進度」
  4. （反覆步驟 2-3，直到功能完成）
  5. 「我覺得暗色模式做好了，幫我合併回 main」
  6. 「合併完成後，幫我刪掉這個分支」
03

Commit 訊息怎麼寫才有用

原文提到 Claude Code 會幫你產生 commit 訊息，這很好用。但如果你想更進一步理解「好的 commit 訊息」長什麼樣子，會幫助你在看歷史紀錄時更快找到需要的版本。

生活比喻
Commit 訊息就像日記的標題。你不會在日記上只寫「今天做了事」，你會寫「跟客戶開會討論 Logo 定稿」。好的 commit 訊息也是一樣，要讓你三個月後回頭看時，不用打開檔案就知道那次改了什麼。

業界常用的格式

很多團隊採用一種叫 Conventional Commits 的格式，用一個「類型」開頭：

# 格式
類型: 簡短描述這次改了什麼

# 實際範例
feat: 新增深色模式切換按鈕
fix: 修復手機版選單無法關閉的問題
style: 調整首頁標題的字體大小和間距
docs: 更新 README 的安裝說明
refactor: 簡化導覽列的程式碼結構
你不需要記住這些格式。跟 Claude Code 說「幫我用 Conventional Commits 格式儲存」，它就會自動幫你分析改動並產生合適的訊息。
04

合併的三種方式

原文提到「合併分支的方式有很多，但我只懂 merge」。讓我用最簡單的方式解釋三種常見的合併方式，這樣你跟 Claude Code 討論時會更有概念：

① Merge（合併）

最直覺的方式。就像兩條路在交叉口匯合，歷史紀錄會保留完整的分支痕跡，你可以清楚看到「這些改動是從哪個分支來的」。

適合什麼時候？
你想保留完整的開發歷程，讓未來回頭看時知道每個功能是怎麼分開開發的。大部分情況用這個就夠了。

② Rebase（變基）

把你的分支「搬到」main 的最前面，讓歷史紀錄看起來像一條直線，好像這些改動一直都是在 main 上面做的一樣。

生活比喻
想像你在寫一篇文章，你把草稿寫在另一張紙上。Merge 是把草稿釘在文章旁邊，Rebase 是把草稿的內容謄寫到文章的最後面，然後把草稿丟掉。成品看起來更乾淨，但你看不出哪些是後來加的。

③ Squash and Merge（壓縮合併）

把分支上的多次 commit 「壓縮」成一個 commit 再合併。適合那種你開發過程中有很多瑣碎的小存檔（像是「修了一個 typo」「忘了存檔」之類的），合併時只留一個乾淨的紀錄。

對 Claude Code 說

「我想把 feature/dark-mode 合併回 main，但我在開發過程中有很多小 commit，可以幫我壓成一個嗎？」
→ Claude Code 就會用 squash merge 幫你處理

不確定用哪種？直接問 Claude Code：「幫我分析一下，這個分支用哪種方式合併比較適合？」它會根據你的 commit 數量和內容給建議。
05

衝突是什麼？不用怕

當你跟另一個人（或你自己在不同分支上）改了同一個檔案的同一個地方，Git 會說「我不知道該用哪個版本」——這就是「衝突（conflict）」。

生活比喻
想像你跟同事各自拿了同一份簡報去改。你把第三頁的標題改成「2025 展望」，同事改成「未來計畫」。當你們要合併時，Git 會問：「第三頁的標題要用哪個？」這就是衝突。Git 不會自作主張幫你選，它會把兩個版本都列出來讓你決定。

衝突不是壞事，它是 Git 在保護你。遇到衝突時：

對 Claude Code 說

「合併的時候好像有衝突了，可以幫我看看是什麼情況嗎？」
→ Claude Code 會列出衝突的檔案和位置，解釋兩邊各改了什麼，然後問你想保留哪個

不要因為怕衝突就不敢開分支。衝突只在合併時才會出現，而且有 Claude Code 在，解決衝突比你想像的簡單很多。
06

Stash：暫時收起來

有時候你正在做 A 功能，突然被要求去修 B 的 Bug。但 A 還沒做完，你不想 commit 一個半成品。這時候就可以用 stash（暫存）。

生活比喻
你正在桌上畫設計稿，突然要去開會。你不想收掉也不想弄亂，就把所有東西推到一邊、蓋上一張紙。開完會回來，再把紙掀開繼續畫。Stash 就是那張紙。
對 Claude Code 說

「我現在做到一半，先幫我暫存起來，我要切到另一個分支去修 Bug」

修完 Bug 之後

「修完了，幫我切回剛才的分支，把暫存的東西拿回來」

07

.gitignore：哪些東西不該被記錄

不是所有東西都需要被 Git 追蹤。有些檔案是你的電腦自動產生的、有些包含機密資訊、有些就是太大了。.gitignore 是一個告訴 Git「請忽略這些檔案」的清單。

常見該忽略的東西

# 套件資料夾（很大，可以隨時重新安裝）
node_modules/

# 環境變數（裡面有密碼、API 金鑰等機密）
.env

# 系統自動產生的檔案
.DS_Store
Thumbs.db

# 編譯產生的檔案（可以隨時重新產生）
dist/
build/
對 Claude Code 說

「幫我建立一個 .gitignore，這是一個 React 專案」
→ Claude Code 會根據你的專案類型，產生一個合適的 .gitignore

特別注意 .env 檔案！如果你的專案有用到任何 API 金鑰或密碼，一定要確認 .env 有在 .gitignore 裡面。一旦推送到 GitHub，你的金鑰就等於公開了。

記住一件事

09

Git 跟 GitHub 的差別

之前我們學了 Git——它是一套裝在你電腦上的版本控制工具，負責追蹤檔案的變更。但 Git 本身只在你自己的電腦上運作。

GitHub 則是一個「雲端平台」，讓你可以把 Git 管理的專案放到網路上。這樣一來，你可以在不同裝置之間同步、把專案分享給別人、或是使用別人公開分享的專案。

生活比喻
Git 就像你手上的相機，負責拍照和整理相簿。GitHub 就像 Google 相簿——讓你把相片放到雲端，可以跟別人共享，別人也可以從上面下載你的照片。相機（Git）跟雲端相簿（GitHub）是兩個獨立的東西，但搭配使用會非常方便。
簡單記法
Git = 你電腦上的版本控制工具
GitHub = 把 Git 專案放到雲端的平台（也是全世界最大的開源社群）
10

Repository（倉庫）：GitHub 的基本單位

在 GitHub 上，每一個專案都存放在一個叫 Repository（簡稱 repo）的地方。你可以把它想成一個雲端資料夾，裡面裝著一個專案的所有檔案和它的完整版本歷史。

以 gstack 為例，它的 repo 就長這樣：

garrytan / gstack
Use Garry Tan's exact Claude Code setup: 23 opinionated tools that serve as CEO, Designer, Eng Manager, Release Manager, Doc Engineer, and QA
⭐ 10,000+ Stars 🍴 Forks 👥 Contributors

一個 repo 頁面上你會看到幾個重要元素：

README.md — 專案的「說明書」。大多數好的 repo 都會有一份 README，告訴你這個專案是什麼、怎麼安裝、怎麼使用。打開 gstack 的 repo，第一眼看到的就是它的 README。

Stars（星星） — 類似社群上的「按讚」。越多 Stars 通常代表越多人覺得這個專案有用。gstack 在發佈後兩天內就突破了一萬顆星星。

Issues — 用來回報問題或提出建議的地方，有點像專案專屬的討論區。

檔案列表 — 就是專案裡的所有檔案和資料夾。你可以直接在 GitHub 上瀏覽它們的內容。

11

Push、Pull、Clone：Git 與 GitHub 之間的溝通方式

Git（你的電腦）和 GitHub（雲端）之間，靠三個動作來交換資料。把這三個搞懂，Git + GitHub 的複合使用就通了。

Git（本地）與 GitHub（雲端）之間的三個核心操作

Clone — 第一次取得專案

在 GitHub 上看到一個想用的專案，你需要把它整份「下載」到你的電腦上。Clone 會把專案的所有檔案連同完整版本歷史一起複製過來。

# 把 gstack 的 repo clone 到你電腦上
git clone https://github.com/garrytan/gstack.git

Clone 只需要做一次。拿到之後，後續的同步就靠 push 和 pull。

Push — 把你的改動「推」到 GitHub 上

你在電腦上做了改動、commit 了存檔之後，這些變更目前只在你的電腦上。Push 就是把這些 commit 「推」到 GitHub 上的 repo，讓雲端也同步到最新狀態。

生活比喻
你在 Word 裡改完文件之後，按了「儲存」只是存在你的電腦上。Push 就像是再多按一個「上傳到 Google Drive」，讓雲端也有最新的版本。

Pull — 把 GitHub 上的最新狀態「拉」回來

如果你在另一台電腦上改過東西（或是有其他人改了），你電腦上的版本就不是最新的了。Pull 就是把 GitHub 上的最新改動「拉」回你的電腦，讓本地同步到最新。

三個動作的簡單記法
Clone = 第一次複製整個專案下來（只做一次）
Push = 把我的改動推到雲端（電腦 → GitHub）
Pull = 把雲端的最新改動拉下來（GitHub → 電腦）
使用 Claude Code 的話，你可以說「幫我把目前的進度推到 GitHub 上」或「幫我拉一下 GitHub 上最新的版本」，不需要記指令。
12

Fork（分叉）：在你的帳號下複製一份

Fork 是 GitHub 上一個很重要的概念，也是你特別問到的。它跟 Clone 很像，但有一個關鍵差別：

CLONE

把別人的 repo 複製到你的電腦上

你拿到的是一份本地的副本

跟原本的 repo 沒有「連結」關係

→ 「我只是想用這個東西」

FORK

把別人的 repo 複製到你的 GitHub 帳號上

你擁有一個獨立的雲端副本

跟原本的 repo 保持「上下游」關係

→ 「我想改東西，而且可能想回饋給原作者」

生活比喻
Clone 是去圖書館影印一本書回家看。Fork 是在圖書館的系統裡「借閱」一本書到你自己的書架上——這本書在你的帳號下，你可以在上面做筆記、改內容。如果你做了很好的註解，你還可以建議圖書館把你的註解加進原版裡面。

什麼時候用 Fork？

想客製化別人的專案 — 假設你覺得 gstack 的某個 Skill 很好，但想根據你自己的工作流程調整，你就可以 fork 一份到你的帳號下，自由修改而不影響原版。

想貢獻改進給原作者 — 如果你在使用過程中發現了 Bug 或想到了改進方式，你可以在你的 fork 裡改好，然後透過「Pull Request」提交給原作者審閱。

Fork 的操作很簡單——在 GitHub 的 repo 頁面右上角，有一個「Fork」按鈕，點下去就完成了。你的帳號下就會多出一個完全一樣的 repo，它下方會標注「forked from garrytan/gstack」。

13

Contributors（貢獻者）是怎麼來的

你在 gstack 的 repo 上看到了 Contributors，好奇他們是不是「一起協作來優化這個專案」——答案是對的，但他們的協作方式跟你想的可能不太一樣。

在開源專案中，Contributors 通常不是被邀請進來的。大部分的流程是這樣的：

  1. 某個人在使用 gstack 時發現了一個 Bug，或覺得某個功能可以做得更好
  2. 他把 gstack 的 repo fork 到自己的帳號下
  3. 在自己的 fork 裡做修改（修 Bug、加功能、改文件）
  4. 改好之後，提交一個 Pull Request 給原作者（Garry Tan）
  5. Garry Tan（或其他維護者）審查這個修改，覺得不錯就「合併」進去
  6. 一旦被合併，這個人就自動成為 Contributor

所以 Contributor 不是一個「被指派的角色」，而是一個「用行動贏得的標籤」。任何人都可以透過 Fork → 修改 → Pull Request 的流程來成為某個專案的 Contributor。

這就是開源的精神
開源專案的原始碼是公開的，任何人都可以看、可以用、可以改。而 GitHub 上的 Fork + Pull Request 機制，就是讓這種「眾人協作」能夠有秩序地進行的制度。原作者保有最終決定權（要不要合併你的修改），貢獻者也不需要取得特別的權限才能參與。
14

Pull Request（PR）：提交你的改動

Pull Request 是 GitHub 上最重要的協作機制。名字聽起來有點奇怪，但概念很簡單：

你在自己的 fork 裡做了改動，然後「請求」原作者把你的改動「拉（pull）」進他的 repo 裡。這個「請求」就是 Pull Request。

生活比喻
想像一本共同編輯的食譜。你覺得某道菜的做法可以改進，但你不能直接去改主廚的版本。你的做法是：先抄一份回家（Fork），在你的版本上改好（修改），然後把你的版本寄給主廚看（Pull Request）。主廚看了覺得不錯，就把你的改法加進正式版本裡（Merge）。

一個 Pull Request 裡通常會包含：

標題和說明 — 你改了什麼、為什麼要改

具體的程式碼改動 — GitHub 會用紅色和綠色標出哪些行被刪除、哪些行被新增

討論區 — 原作者和其他人可以在這裡提問、給建議、要求修改

Pull Request 不是「丟出去就結束」。它更像是一場對話——原作者可能會說「這邊可以再改一下」，你就回去修改後再更新。直到雙方都滿意了，才會合併進去。

15

Vibe Coding 時的 Git + GitHub 實務流程

了解了個別的操作之後，來看看 Vibe Coding 時，整個流程串起來會長什麼樣子：

Vibe Coding 的日常節奏：commit（存檔）→ push（同步到雲端）→ pull（在另一台裝置繼續）

最核心的節奏就是 commit → push 這個循環。Commit 是在你電腦上存檔，push 是同步到雲端。養成「做完一個小段落就 commit + push」的習慣，你的專案就永遠有雲端備份。

什麼時候要 push？
沒有硬性規定，但幾個好的時機：做完一個功能的時候、一天結束要離開電腦前、重要的改動之後。簡單說，只要你覺得「這個進度不想丟失」，就 push 一下。

如果你要嘗試風險比較大的改動，流程會多一步：先開分支，在分支上開發，確認沒問題後合併回 main，最後 push 到 GitHub。

整個流程你都可以用自然語言跟 Claude Code 說。它知道怎麼把「幫我推到 GitHub」翻譯成 git push、把「幫我從 GitHub 拉最新的」翻譯成 git pull。
16

實戰：使用 gstack 的 Skill

回到你最初的問題：如果想使用 gstack 裡的某個特定 Skill，你可以怎麼做？

情境 A：直接使用整套 gstack

gstack 有提供一鍵安裝的指令。你只需要在 Claude Code 裡貼上安裝指令，它就會自動 clone 整個 repo 到你電腦上，然後執行設定。

# gstack 的官方安裝方式
git clone --single-branch --depth 1 https://github.com/garrytan/gstack.git ~/.claude/skills/gstack
cd ~/.claude/skills/gstack && ./setup

安裝後，gstack 的檔案會存放在 ~/.claude/skills/gstack/ 這個路徑。~ 代表你電腦的使用者主目錄，.claude 是 Claude Code 的設定資料夾（前面的小數點代表它是一個隱藏資料夾）。所以它其實就在你電腦裡 Claude Code 自己的 Skill 目錄下。

這裡有兩個你可能沒見過的參數：

--single-branch 只下載主要分支，不需要所有分支的歷史

--depth 1 只下載最新一次的 commit，不需要完整歷史。這兩個加在一起，讓下載速度快很多。

你可以直接跟 Claude Code 說「幫我安裝 gstack」，然後把 GitHub 連結貼給它。Claude Code 會幫你跑完上面這些步驟。

情境 B：只想參考某個特定 Skill 的寫法

如果你不想安裝整套 gstack，只想看某個 Skill（比如 /qa 或 /design-review）是怎麼寫的，你有幾個做法：

直接在 GitHub 上瀏覽 — 打開 gstack 的 repo，點進對應的資料夾（例如 qa/SKILL.md），就能直接在網頁上看到完整的 Skill 內容。不需要下載任何東西。

Clone 下來本地研究 — 如果你想更仔細地研究多個 Skill 的結構，可以整個 clone 下來慢慢看。

Git 的 clone 不能只抓 repo 裡的某個資料夾。Git 的設計是以整個 repo 為單位來管理的，所以你只能 clone 整份，不能指定「只要 qa 這個資料夾」。不過如果只是想看某個檔案的內容，直接在 GitHub 網頁上瀏覽是最快的方式。

情境 C：想在 gstack 的基礎上做自己的版本

如果你覺得 gstack 的某個 Skill 很好，但想根據自己的需求調整，這時候就適合用 Fork：

  1. 在 GitHub 上按 Fork，複製一份到你的帳號下
  2. Clone 你自己的 fork 到電腦上
  3. 修改你想調整的 Skill 檔案
  4. Commit 並 push 到你自己的 GitHub repo 上

這樣你就有了一份屬於你自己的 gstack，可以自由客製化。而且因為是 Fork，未來原版更新時，你也可以把最新的改動同步過來。（會不會覆蓋你的修改？下一節解答。）

回顧整個流程
GitHub 的核心操作其實就是這幾個動作的排列組合：

只想用 → Clone 下來就好
想改、想客製 → 先 Fork，再 Clone 你的 fork
想回饋給原作者 → Fork → 修改 → 發 Pull Request
17

Fork 同步更新時，不會覆蓋我的修改嗎

這是一個很直覺的擔心，但答案是：通常不會。

當你從原版同步更新到你的 fork 時，Git 會自動進行「合併」——它會把原版的新改動和你自己的修改整合在一起，而不是用原版去覆蓋你的東西。

生活比喻
想像你跟原作者各自在編輯同一本書的不同章節。你改了第三章，原作者新增了第七章。同步更新時，Git 會說：「第三章用你的版本，第七章用原作者新增的」，兩邊的修改都保留。

只有一種情況會需要你手動介入：當你跟原作者改了同一個檔案的同一個地方。這時候 Git 會跟你說「這裡有衝突，你想用哪個版本？」讓你自己決定。這就是我們之前學過的「衝突」。

三種情況
你改了 A 檔案，原版改了 B 檔案 → 完全不衝突，兩邊的改動都自動保留

你跟原版都改了同一個檔案，但改的是不同段落 → Git 通常也能自動合併

你跟原版改了同一個檔案的同一段 → Git 會提醒你有衝突，讓你選擇保留哪個版本

在實務上，如果你 fork 的是一個大型專案（像 gstack 有幾十個 Skill 檔案），而你只改了其中一兩個，跟原版衝突的機率非常低。大部分時候同步更新都是自動、無痛的。

遇到衝突也不用擔心。跟 Claude Code 說「幫我同步原版的最新更新」，如果有衝突，它會列出來讓你決定怎麼處理。
